# TelegArm — Lessons Learned (technical)

Concrete gotchas discovered while building this app. Each: **problem → fix**.

## WinForms / MaterialSkin
- **MaterialSkin overrides `Control.Font` after construction.** → Store every font used in OnPaint in a private readonly field; never read `this.Font`.
- **`SetProcessDpiAwareness(2)` (per-monitor) + manual `_scale = DpiX/96` = double scaling / blur.** MaterialSkin.2 has no per-monitor DPI. → Use `SetProcessDpiAwareness(1)` (system) + `AutoScaleMode.Font` on all forms; no manual DPI math; fixed logical sizes.
- **`SplitContainer` throws if `SplitterDistance` is set before it has a valid width** (e.g. in an object initializer while width is the default 150). → Add it to the form first, then set `Panel1MinSize`/`Panel2MinSize`/`SplitterDistance` (guarded in try/catch).
- **`FlowLayoutPanel.Scroll` does not fire reliably on mouse wheel** (NewValue != 0). → Add a `MouseWheel` handler and check the position in a `BeginInvoke` (runs after the wheel scroll applies): `VerticalScroll.Value <= SmallChange*3`.
- **`ScrollControlIntoView(last)` stops flush against the edge and ignores bottom padding.** → Scroll to true bottom: `PerformLayout()` then `AutoScrollPosition = new Point(0, VerticalScroll.Maximum)`.
- **`Controls.Clear()` does NOT dispose children** → owner-painted controls keep running timers / leak GDI. → Dispose children explicitly when clearing a panel (e.g. `ClearMessagePanel`).
- **An owner-painted bubble that spans the full panel width (for left/right alignment) must hit-test clicks against the *painted* region, not the whole control** — otherwise clicking the empty track beside a right-aligned card still "hits" it. → Record the painted rect per paint path (`_imageRect`/`_fileRect`/`_bodyRect`) and check `rect.Contains(e.Location)` in `OnMouseClick`.
- **WinForms `Button` can't render color emoji (GDI); glyphs vary by font/OS.** → Draw icons with GDI+ shapes in a custom `Control.OnPaint` (see `MediaControlButton`). Don't rely on `Segoe UI Symbol` glyphs for cross-OS UI (esp. RT 8.1).
- **`TableLayoutPanel` is the deterministic way** to lay out edge+fill regions (avoids dock z-order ambiguity). Use it for toolbars/bars/panels.
- **Owner-painted CHILD controls don't render inside a toolbar `TableLayoutPanel` row** — and native `Button` *text* often won't paint there either (you get blank/white boxes); only native `Label`/`TrackBar` reliably show. Yet owner-painted controls render fine in the message `FlowLayoutPanel` (e.g. the bubbles). → For a control that lives in a toolbar row, **paint the WHOLE control yourself** in its own `OnPaint` (no child controls) and hit-test clicks by region. This is what fixed the mini player bar (`MiniPlayerBar`) and is the bubble pattern applied to a bar.
- **Collapsing/expanding a `TableLayoutPanel` row resizes the child `FlowLayoutPanel`, which resets its AutoScroll to the top.** → Capture `-panel.AutoScrollPosition.Y` before the row-height change and restore `panel.AutoScrollPosition = new Point(0, y)` after (see `MainForm.UpdateMiniBar`).
- **Reserve the scrollbar width when a top-docked bar shares a column with a scrolling panel** so its right-edge controls don't sit over the scrollbar: `SystemInformation.VerticalScrollBarWidth + margin` (DPI-safe).
- **Setting `Visible = false` in a `UserControl` constructor fires `OnVisibleChanged` BEFORE your fields are initialized** → NullReferenceException on startup. Guard the override (`if (_timer == null) return;`).
- **A method named `Refresh()` on a Control hides `Control.Refresh()` (CS0114).** → Name custom repaint helpers something else (`UpdateUi`/`OnTick`).
- **`BackColor = Color.Transparent` on a control that does `g.Clear(BackColor)` (e.g. `MaterialLabel`, a custom `Control`) paints BLACK** on a normal (non-layered) window — "transparent" isn't real transparency here. → For themed bars/strips, use an **owner-painted single control** that `g.Clear`s a solid themed color (the `VideoControlBar`/`MiniPlayerBar`/`SelectionBar` pattern), not child controls with transparent backgrounds.
- **`MediaControlButton` has a hard-coded dark circle** (built for the dark video bar) — reusing it on a themed/light bar shows black blobs. → Draw bar buttons as themed GDI+ accent circles in the bar's own `OnPaint`.
- **A client's own outgoing message isn't reliably delivered via `OnUpdates`** (it may only come back as the send call's RPC result). → Update optimistic UI (chat-list row, bubbles) explicitly at send time; don't wait for an echo.
- **For a context menu that must also work with touch, handle `WM_CONTEXTMENU` (0x007B) in `WndProc` — don't hook `MouseUp`/right-click.** Windows synthesizes `WM_CONTEXTMENU` for right-click, the keyboard menu key, AND touch-and-hold, so one override covers mouse + touch on the tablet for free. `lParam` is the screen point packed as signed shorts (low=x, high=y), or `-1` for the keyboard key (use the control's center then). This is how the message-bubble menu works on the Surface RT without any extra gesture code.

## Audio (LibVLCSharp.Shared, audio-only)
- **LibVLC ignores `MediaPlayer.Volume`/`Mute` set right after `Play()`** — the audio output isn't created until playback actually starts, so the clip plays muted. → Subscribe `MediaPlayer.Playing` and (re)apply `Volume`/`Mute`/`SetRate` there (and again right after `Play(media)`).
- **One static audio-only `MediaPlayer` (no `VideoView`) is enough** for voice/audio; keep a per-chat playlist `(id,title)` + a `resolve(id)→Task<string>` so next/previous can download on demand. Raise a single `StateChanged` event and let each view (bubbles + mini bar) re-render from the player's current state.

## Native video (LibVLCSharp.WinForms)
- **The native VLC video window repaints over owner-painted WinForms controls** that share the same control/region; native controls (TrackBar) survive, custom-painted ones (buttons/seek) go blank during playback. → Two ways out, and the **current** code uses the second: (1) build the bar from native controls only (flat Buttons + TrackBars + Labels); (2) **own-paint a single control placed in a row BELOW the video (not overlapping it) AND `Invalidate()` it every timer tick** so it repaints over any VLC bleed. Neither relocating the bar *nor* per-tick `Invalidate` worked **alone**; placing it below the surface **and** ticking the invalidate together make owner-painting viable (this is how `VideoControlBar` replaced the native bar).
- **VLC native arch must match the PROCESS arch** (x86 process → x86 VLC; ARM32 process → ARM32 VLC). → Detect via folder presence; wrap `Core.Initialize` in try/catch (catch `BadImageFormatException`) and fall back. Show `Environment.Is64BitProcess` to the user.
- **`Core.Initialize()` must be called exactly once per process.** → `VlcEnvironment.TryInitialize()` guards with a static flag + try/catch.
- **Dispose order matters:** stop → `videoView.MediaPlayer = null` → `MediaPlayer.Dispose` → `LibVLC.Dispose` → `VideoView.Dispose`.

## WTelegramClient
- **The update event is `OnUpdates`, not `OnUpdate`** (`Func<UpdatesBase,Task>`); also `OnOwnUpdates`. Walk `updates.UpdateList` for `UpdateNewMessage` (covers short-message forms).
- **Config callback must always be able to return `phone_number`.** Returning null aborts `LoginUserIfNeeded`. With a valid session, supplying the phone logs in **without** a code. → Persist + supply the phone for silent resume.
- **Source of truth for "logged in" is `Client.User != null`** (`IsAuthorized`), not the absence of exceptions.
- **AUTH_KEY_DUPLICATED:** two clients on one session file → Telegram revokes it. → Single-instance mutex; don't open the same session twice.
- **`InputMessagesFilterEmpty` / `InputPeerEmpty` are not concrete types** in WTelegramClient — pass `null` (it serializes the "empty" constructor). `Messages_SearchGlobal(query, limit:...)` works with filter/offset_peer omitted.
- **`DownloadFileAsync(doc, stream, null, progress)` is ambiguous** (PhotoSizeBase vs VideoSize overloads). → Cast `(PhotoSizeBase)null`.
- **`Photo.LargestPhotoSize`** (PhotoSizeBase: Width/Height/FileSize/Type) and the `DownloadFileAsync(Photo, stream, PhotoSizeBase)` overload avoid building `InputPhotoFileLocation` by hand.
- **`User.ToInputPeer()` / `ChatBase.ToInputPeer()`** convert resolved peers to `InputPeer`.
- **Replying is just a named arg on the send helper:** `SendMessageAsync(peer, text, reply_to_msg_id: id)` (id is the message `ID`, an int). No separate reply RPC.
- **Reactions:** `Messages_SendReaction(peer, msg_id, Reaction[])` — pass `new[]{ new ReactionEmoji{ emoticon } }` to set, `null` to clear. Read counts from `Message.reactions.results` (`ReactionCount`); **"did I react" is the flag `ReactionCount.Flags.has_chosen_order`**, NOT `chosen_order != null` (that field is a non-nullable `int`). `ReactionCustomEmoji` has no static glyph.
- **Chat folders:** `Messages_GetDialogFilters()` → `Messages_DialogFilters.filters` (`DialogFilterBase[]`). Concrete `DialogFilter` has `flags` (contacts/non_contacts/groups/broadcasts/bots/exclude_muted/exclude_read/…), `include_peers`/`pinned_peers`/`exclude_peers` (`InputPeer[]`). **`DialogFilterBase.Title` is a `TextWithEntities`, not a string → use `.Title?.text`**, and its getter can throw on some entries (guard with try/catch). There is **no** `DialogFilterDefault` type in 4.4.6 — synthesize the "All" tab yourself.
- **Stickers:** `Messages_GetAllStickers` → `Messages_AllStickers.sets` (`StickerSet[]`; `.title`/`.id`/`.access_hash`); `Messages_GetStickerSet(new InputStickerSetID{ id, access_hash })` → `.documents`; `Messages_GetStickers(emoticon)` → `Messages_Stickers.stickers` for emoji→sticker search.

## ARM32 / native libs
- **SkiaSharp 2.88.x ships `win-x86`/`win-x64`/`win-arm64` natives but NOT `win-arm` (32-bit)** — so SkiaSharp.Skottie (Lottie) `DllNotFound`s on Surface RT / Win10 ARM32. The real `.tgs` path is **rlottie** (Samsung C++) via P/Invoke, cross-compiled to an ARM32 Windows `rlottie.dll` (`cmake -A ARM -DBUILD_SHARED_LIBS=ON`). Its C API (`inc/rlottie_capi.h`) is undecorated cdecl: `lottie_animation_from_data/_get_totalframe/_get_framerate/_render/_destroy`. Output buffer is BGRA premultiplied → matches GDI `Format32bppPArgb` (no conversion).
- **A native DLL is architecture-specific; "AnyCPU" only makes the *managed* assembly neutral.** Ship per-arch (`rlottie\x86\`, `rlottie\ARM\`) and `SetDllDirectory` the right one at startup based on `PROCESSOR_ARCHITECTURE` (x86 process reports "x86" even under WOW64; ARM32 reports "ARM"). Then `[DllImport("rlottie")]` resolves it.
- **A VS-built DLL links the dynamic CRT (`msvcp140`/`vcruntime140` + UCRT `api-ms-win-crt-*`), which Windows RT 8.1 lacks** (no ARM32 VC++ redist exists). Symptom: "msvcp140.dll is missing" loader popup, or silent `DllNotFound`. Fixes: rebuild with **static CRT** (`-DCMAKE_MSVC_RUNTIME_LIBRARY=MultiThreaded` + `-DCMAKE_POLICY_DEFAULT_CMP0091=NEW`, ideally `-DCMAKE_SYSTEM_VERSION=6.3` for the 8.1 SDK), or ship the CRT DLLs beside the exe. Call `SetErrorMode(SEM_FAILCRITICALERRORS|SEM_NOOPENFILEERRORBOX)` at startup so a missing optional dep fails silently into the fallback instead of popping the loader dialog. Diagnose with an explicit `LoadLibraryEx` + `Marshal.GetLastWin32Error` (126 = missing dep, 193 = wrong arch, 127 = missing export).
- **DPAPI (`System.Security.Cryptography.ProtectedData`)** works on all Windows incl. ARM (needs a `System.Security` assembly reference). Used to encrypt the persisted phone; degrade to plaintext on failure.
- **WinForms dock order is the reverse of add order** (last added docks first). A `Fill` child must be added **before** its `Top`/`Bottom` siblings to receive the leftover space.

## Scrollbars / theming
- **`SetWindowTheme(h, "DarkMode_Explorer", null)` only themes scrollbars on Windows 10 1809+** — it's a silent no-op on RT 8.1, which has no dark scrollbar visual style. For a consistent look everywhere, replace native bars with a **custom owner-painted scrollbar** (`UI/Controls/ThemedScrollBar.cs`, vertical *and* horizontal): hide the native bar with `ShowScrollBar(SB_VERT/SB_HORZ, false)` (re-hide on Scroll/Layout/ClientSizeChanged), and drive `AutoScrollPosition`. Children of an AutoScroll panel scroll with content, so the bar can't live inside it — **wrap** the panel (Fill) + bar (Right/Bottom) in a plain host Panel.
- **A WinForms `Timer` ticks on the UI thread**, so an animation loop (e.g. Lottie frames) that renders into a reused `Bitmap` and the paint that reads it never overlap — no locking, no per-frame allocation, no tearing. One shared static timer driving N animators beats N timers.
- **Deleting branches by peer kind:** channels/supergroups use `Channels_DeleteMessages(InputChannel, ids)`; users/basic groups use `Messages_DeleteMessages(ids, revoke)` where `revoke = true` deletes for everyone (`false` = just for me). Build the `InputChannel` from an `InputPeerChannel`'s `channel_id` + `access_hash` (same pattern as `ReadHistoryAsync`).
- **`Messages_ForwardMessages(from_peer, id[], random_id[], to_peer)` requires a `random_id` array the same length as `id[]`** — one unique long per forwarded message (generate them yourself). Returns `UpdatesBase`.
- **Sending media mirrors sending text:** `client.UploadFileAsync(path, progress)` returns an `InputFileBase`; wrap it in `InputMediaUploadedPhoto`/`InputMediaUploadedDocument` and call `client.SendMessageAsync(peer, caption, media)` — the same helper as text, which handles `random_id` internally and returns the confirmed `Message`. No need to build `Messages_SendMedia` by hand.
- **`UploadFileAsync(path, …)` streams from disk** — pass the path, never pre-read the file into a `byte[]` (this is the ARM32-safe upload path; only small images are first compressed to a temp file).
- **`DocumentAttributeVideo.duration` is `double`** (it changed across TL layers); `w`/`h` are `int`. Set `flags = supports_streaming` to send a streamable video document **without re-encoding** (we deliberately don't probe duration/w/h, which would require decoding the file on ARM32).
- **`InputMediaUploadedDocument.force_file`** makes Telegram send the upload as a plain file instead of auto-detecting it as media (used for "Send as file").
- **Optimistic outgoing bubbles can't be reconciled by message id** — the local placeholder has no server id yet, and the echo (`UpdateNewMessage`) can arrive **before** the send call returns, because a modal send dialog (`ShowDialog`) keeps pumping the update loop during the upload `await`. So the success callback and the echo race, and matching the echo by id never hits the negative temp id → duplicate bubble + "needs reload". → **Reconcile from whichever path fires first and make the other idempotent.** Both `HandleIncomingMessage` (for an outgoing echo with media) and the dialog's success callback perform the *same* in-place swap (set real id, clear pending, add real id to the dedupe set, `PerformLayout`+`Invalidate`); the second to run no-ops via a guard (`MessageId >= 0`). This is sufficient **only because uploads are sequential** (one pending bubble in flight), making the match unambiguous. If sends ever go concurrent, switch the match key to a **per-send client `random_id`** instead of "the single pending bubble". (See `TrySwapPendingBubble` in MainForm.)
- **A caption shown on an optimistic media bubble should match the post-reload render, not add a placeholder.** The bubble text is `readonly` (set at construction), so plumb the caption through at creation time. With no caption, leave it **empty** — a photo/video/file with no caption renders with no text after reload, so a "📷 Photo" placeholder would just flash and disappear. (Bonus: a caption flows through the same RTL detection as normal text.)

## C# / build
- **`using TL;` makes `Message` ambiguous** with `System.Windows.Forms.Message`. → Use `using Message = TL.Message;` and fully-qualify `ProcessCmdKey(ref System.Windows.Forms.Message msg, …)`.
- **LangVersion defaulted to C# 7.0** (no `default` literal / target-typed). → `<LangVersion>7.3</LangVersion>`.
- **LibVLCSharp 3.x ships net40 libs** → consumable from .NET 4.7 (NuGet picks net40; net471 excluded as > 4.7).
- **.NET 4.7 is the max** that ships on Windows 10 ARM32 build 15035. Targeting 4.7 needs the 4.7 targeting pack on the build box; after retargeting, re-run `nuget restore` (obj/project.assets.json caches the old TFM).
- **Pure-managed NuGet packages sidestep the ARM32 native-binary problem.** NAudio (capture) + Concentus/Concentus.OggFile (Opus/OGG) are AnyCPU IL → they JIT to ARM32 with no per-arch native DLL to source (unlike VLC/ffmpeg). **Watch the TFM, though:** NAudio 2.x needs net4.7.2 → use **NAudio 1.10.0** (net35) on net4.7. Concentus 1.1.7 is netstandard1.1 (works, pulls a System.* facade tree). Opus encoding in pure C# is heavier than native but fine for mono low-bitrate voice on ARM32.
- **Voice notes = OGG/Opus + `DocumentAttributeAudio{ voice }`** (`audio/ogg`). Pre-seed the audio cache (copy the temp OGG to the voice cache path) so the just-sent bubble plays instantly.
- **No nuget.exe/dotnet on this build machine** → downloaded nuget.exe to repo root; restore with `-MSBuildPath "D:\Program Files\Vscom\MSBuild\15.0\Bin"`; build with that MSBuild.

## Theming
- **Resolve light/dark in ONE place** (`ThemeHelper.IsDark`) so a user override (Light/Dark) and System (OS-following) can't drift between forms. Forms should read `IsDark`, never `IsDarkMode()` directly.
- **Separate "accent changed" from "light/dark changed"** in the `UserPreferenceChanged` handler: accent (Color category) should always follow the OS, but the OS light/dark switch (General) must only move the app when `Mode == System` — otherwise a manual override gets clobbered on the next OS theme change.
- **Apply the persisted mode before any window is created** (`ThemeHelper.InitMode` in `Program.Run`) so the login/main form is correct on the first frame, with no flash.
- **A themed control built before the palette exists** (e.g. native bar buttons created in `BuildUi`, before `ApplyTheme`): capture references and re-color them in `ApplyTheme` (call it once right after `BuildUi`, and again on `ThemeChanged`).

## DPI + multi-monitor (reference, for tray/global features later)
- `Screen.AllScreens` returns logical coords; low-level mouse hook (`WH_MOUSE_LL`) returns physical pixels. With `SetProcessDpiAwareness(1)` they align at system DPI. (Relevant if/when adding a tray icon or global hooks.)

## Image/GDI
- `Image.FromStream` keeps the stream open lazily → copy to a `new Bitmap(img)` so the stream can be disposed. Cache owns Images; owner-painted controls reference them and must NOT dispose them.
